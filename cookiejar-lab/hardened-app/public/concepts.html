What Are Cookies

The foundational technology behind web sessions - from 1994 to today.

History & Origin

Lou Montulli at Netscape invented cookies in 1994 to solve the "shopping cart problem" - HTTP is stateless, so there was no way to remember items between page loads.

The name comes from "magic cookie" - a Unix/computing term for an opaque data token passed between programs.

Originally specified in RFC 2109 (1997), updated in RFC 2965, and finalized as RFC 6265 (2011) - the current standard.

Cookies predated JavaScript by one year - they're a native HTTP feature, not a JavaScript feature.

How Cookies Work in HTTP

Server sends Set-Cookie: session=abc123; Path=/; HttpOnly in the HTTP response headers.

Browser stores the cookie and automatically attaches it to every subsequent request to that domain via the Cookie: session=abc123 header.

This is automatic browser behavior - no JavaScript is needed.

Cookies are scoped by domain and path - a cookie set for example.com won't be sent to other.com.

Maximum cookie size: ~4KB per cookie, ~180 cookies per domain.

  Browser                         Server
     |                              |
     |  POST /login (credentials)   |
     |----------------------------->|
     |                              |
     |  200 OK                      |
     |  Set-Cookie: session=xyz     |
     |<-----------------------------|
     |                              |
     |  GET /dashboard              |
     |  Cookie: session=xyz         |
     |----------------------------->|
     |                              |
     |  200 OK (authenticated)      |
     |<-----------------------------|

Types of Cookies

Type Lifetime Created By Privacy Impact
Session Cookie Until browser closes Any domain Low - temporary
Persistent Cookie Set expiry date (days/years) Any domain Medium - survives restart
First-Party Cookie Varies Same domain as URL bar Low - expected behavior
Third-Party Cookie Varies Different domain (ads, trackers) High - cross-site tracking
Note: Third-party cookies are being phased out. Chrome's Privacy Sandbox, Firefox's Total Cookie Protection, and Safari's ITP all restrict them.
Security implication: Session cookies vanish when browser closes, but infostealers read the cookie store file while the browser is open - "session only" provides no protection against malware.

Cookie Attributes Reference

Attribute Example Purpose
Domain Domain=example.com Which domains receive the cookie. Omit for same-origin only.
Path Path=/api URL path scope. / means the entire site.
Expires Expires=Thu, 01 Jan 2026 00:00:00 GMT Absolute expiration date.
Max-Age Max-Age=86400 Seconds until expiry (overrides Expires).
Secure Secure Only sent over HTTPS connections.
HttpOnly HttpOnly Inaccessible to JavaScript (document.cookie).
SameSite SameSite=Strict Controls cross-site sending. Strict/Lax/None.
__Host- prefix __Host-session=abc Must be Secure, Path=/, no Domain - strongest binding.
__Secure- prefix __Secure-session=abc Must have Secure flag. Weaker than __Host-.
Partitioned Partitioned CHIPS - isolates cookie per top-level site. Part of Privacy Sandbox.

Where Cookies Are Stored

Browsers store cookies in SQLite databases on disk:

  • Chrome: ~/Library/Application Support/Google/Chrome/Default/Cookies (macOS), %LOCALAPPDATA%\Google\Chrome\User Data\Default\Cookies (Windows)
  • Firefox: <profile>/cookies.sqlite
  • Safari: ~/Library/Cookies/Cookies.binarycookies

Infostealers (RedLine, Raccoon, Vidar, Lumma) read these SQLite files directly - no browser interaction needed.

Chrome added App-Bound Encryption (v127+) to protect cookies at rest, but infostealers quickly adapted by running code within the browser process context.

What Is JWT

JSON Web Tokens - the dominant format for modern session tokens and API credentials.

JWT Structure

A JWT has three Base64URL-encoded parts separated by dots: header.payload.signature

Header
{"alg": "HS256", "typ": "JWT"}
Payload
{"sub": "1234", "name": "alice", "exp": 1711500000}
Signature
HMAC-SHA256(base64url(header) + "." + base64url(payload), secret)
  • Header - algorithm and token type
  • Payload - claims (data)
  • Signature - verifies integrity and authenticity
CRITICAL: Encoding ≠ Encryption. Anyone can decode a JWT's header and payload - they're Base64URL-encoded, not encrypted. The signature only proves the token wasn't tampered with.

How JWT Authentication Works

Step-by-step flow:

  1. Client sends credentials (username/password, OAuth token, etc.)
  2. Server verifies credentials against its database
  3. Server creates JWT with user data, signs it with a secret key
  4. Server sends JWT to client (usually as a cookie or in a response body)
  5. Client sends JWT with every request (Cookie header or Authorization: Bearer)
  6. Server decodes JWT, verifies signature, checks expiry - grants or denies access
Contrast with opaque tokens: An opaque token (like sess_a8f3k2j) has no data - the server must look it up in a database. JWTs are self-contained - the server can verify them without a database lookup (stateless verification).

JWT Claims Reference

Registered claims from RFC 7519:

Claim Name Purpose Example
iss Issuer Who created the token "https://auth.example.com"
sub Subject Who the token is about "user_12345"
aud Audience Who the token is for "https://api.example.com"
exp Expiration When the token expires 1711500000 (Unix timestamp)
nbf Not Before When the token becomes valid 1711400000
iat Issued At When the token was created 1711400000
jti JWT ID Unique token identifier "abc-123-def"

Custom claims: Applications can add any custom claims (userId, email, role, etc.). Keep payloads small - every claim adds to token size, and tokens are sent with every request.

Signing Algorithms

Algorithm Type Key Use Case
HS256 Symmetric Shared secret Single server apps. This lab uses HS256.
RS256 Asymmetric RSA key pair Microservices - public key can verify without the private key.
ES256 Asymmetric ECDSA key pair Smaller keys, same security as RS256. Mobile-friendly.
EdDSA Asymmetric Ed25519 key pair Modern, fast, compact. Growing adoption.
The alg: "none" attack: Early JWT libraries accepted tokens with "alg": "none" - meaning no signature at all. An attacker could forge any token by setting the algorithm to "none" and stripping the signature. Modern libraries reject alg: none by default, but it remains a classic example of why you must validate JWT configuration server-side.

JWT Footprint in Modern Apps

JWT is everywhere. Understanding its use cases helps you assess attack surfaces:

  • Web Sessions: The most visible use. JWTs store user identity in session cookies (exactly what this lab demonstrates).
  • Mobile Authentication: Mobile apps store JWTs in secure storage (iOS Keychain, Android Keystore) and send them as Bearer tokens.
  • Microservices Communication: Service A calls Service B with a JWT. Service B verifies the signature without calling an auth server - critical for low-latency service-to-service calls.
  • OAuth 2.0 / OIDC: OAuth access tokens are often JWTs. OIDC ID tokens are always JWTs. When you "Sign in with Google," the ID token proving your identity is a JWT.
  • API Gateways: Kong, AWS API Gateway, Cloudflare Access - all validate JWTs at the edge before requests reach your backend.
  • SSO (Single Sign-On): Okta, Auth0, Azure AD issue JWTs. One login, many applications - each app verifies the same JWT.
  • Serverless Authorizers: AWS Lambda authorizers, Cloudflare Workers - validate JWTs without a running server.
  • Webhook Verification: Services sign webhook payloads with JWTs so recipients can verify authenticity.

Common JWT Pitfalls

  1. Storing secrets in the payload - JWTs are not encrypted. Anything in the payload is visible to anyone who decodes it. Never put passwords, API keys, or PII in JWT payloads.
  2. No revocation mechanism - A stateless JWT is valid until it expires. You can't "log out" a stolen JWT. Mitigation: short expiry + refresh tokens + server-side session store.
  3. Token bloat - Every claim adds bytes. JWTs are sent with every request. A 2KB JWT in a cookie means 2KB of extra data on every single HTTP request.
  4. Algorithm confusion attacks - If the server accepts both HS256 (symmetric) and RS256 (asymmetric), an attacker can sign a forged token with the RS256 public key using HS256. Modern libraries mitigate this by requiring explicit algorithm configuration.
  5. Long expiry times - A 7-day JWT (like this lab's vulnerable app) gives attackers a huge replay window. Use short-lived tokens (15 minutes or less) with refresh token rotation.
  6. Weak signing keys - HS256 with a short or guessable secret can be brute-forced. Use cryptographically random keys of at least 256 bits.

Session Models Compared

Six approaches to managing user sessions - each with different trade-offs in security, scalability, and complexity.

Stateless JWT

Token contains all session data. Server verifies signature only - no database lookup.

Used in: This lab's vulnerable app. Many SPAs. Early Node.js/Express tutorials.

Pros: Horizontally scalable (any server can verify), no session store infrastructure, fast verification.
Cons: Cannot revoke individual sessions (token valid until expiry), large tokens, payload visible to client.
Security: Stolen token = full access until expiry. No way to invalidate on logout or password change. 7-day tokens are common and dangerous.

Verdict: Simple but dangerous as sole session mechanism. Fine for short-lived API tokens; risky for web sessions.

Server-Side Sessions (Opaque Tokens)

Server generates a random session ID, stores session data in a database. Cookie contains only the opaque ID.

Used in: Django, Rails, PHP (default), Express + express-session, Laravel.

Pros: Instant revocation (delete session from store), tiny cookies (~32 bytes), no sensitive data in cookie, full server control.
Cons: Every request requires a session store lookup (adds latency), session store is a single point of failure, horizontal scaling requires shared store (Redis, database).
Security: Stolen session ID still grants access, BUT: server can revoke immediately on logout, can enforce concurrent session limits, can detect anomalies server-side.

Verdict: The tried-and-true approach. Higher operational overhead but much better security posture than stateless JWT.

JWT + Server-Side Hybrid

Short-lived JWT (15 min) for request authentication + server-side session store for revocation + refresh token rotation for renewal.

Used in: This lab's hardened app. Auth0, Firebase Auth, many production systems.

Pros: Fast verification (JWT signature check), revocable (server-side store), short replay window, refresh rotation detects theft.
Cons: More complex to implement correctly, still requires session store, refresh token logic adds code.
Security: Multiple defense layers. Stolen access token expires in minutes. Stolen refresh token triggers rotation detection. Server-side revocation as backstop.

Verdict: Best of both worlds for most production web applications. The approach this lab recommends.

Token-Bound Sessions (DBSC)

Device Bound Session Credentials - cryptographically binds sessions to hardware using the device's TPM.

How it works: The browser generates a key pair in the TPM. The public key is registered with the server. Every session request includes a proof-of-possession signed by the private key. The private key never leaves the hardware.

Status: Chrome Origin Trial. Not yet widely available.

Pros: Stolen cookie is useless without the hardware key. Defeats all infostealer attacks that exfiltrate cookies.
Cons: Requires browser support (Chrome only for now), requires server-side changes, TPM availability varies, doesn't work across devices.
Security: The strongest protection against session theft. Eliminates the entire class of cookie replay attacks.

Verdict: The future of session security. Watch for browser adoption.

Backend-for-Frontend (BFF)

A server-side proxy between the browser and the API. The BFF holds the actual access/refresh tokens. Tokens never reach the browser.

Used in: Duende BFF, Azure AD + MSAL.js with proxy, many enterprise SPAs.

Pros: Tokens never exposed to client-side JavaScript or browser storage. Eliminates client-side token management. XSS can't steal what isn't there.
Cons: Additional infrastructure (proxy server), adds latency per request, more complex deployment.
Security: Very strong for SPAs. The browser only sees an httpOnly session cookie to the BFF. Even if XSS runs on the page, there are no tokens to steal.

Verdict: Excellent choice for SPAs that need OAuth tokens. Eliminates the "where do I store my token?" problem.

Mutual TLS (mTLS)

Both the server AND the client present TLS certificates. The client certificate proves device identity at the transport layer.

Used in: Service-to-service (microservices mesh), banking/financial APIs, corporate VPNs, IoT devices.

Pros: Strongest device binding - certificate theft requires private key extraction. No cookies or tokens involved. Works at the TLS layer (below HTTP).
Cons: Certificate management overhead (issuance, rotation, revocation), impractical for browsers (users can't manage certificates), CRL/OCSP infrastructure needed.
Security: Extremely strong for controlled environments. A stolen session is useless without the client certificate's private key.

Verdict: Ideal for service-to-service communication. Not practical for browser-based consumer web apps.

Side-by-Side Comparison

How each session model stacks up across key dimensions.

Model Revocable? Scalable? Theft Protection Complexity Best For
Stateless JWT No (until expiry) Excellent None Low Short-lived API tokens
Server-Side Sessions Yes (instant) Moderate (needs shared store) Revocation only Low-Medium Traditional web apps
JWT + Server Hybrid Yes Good Multi-layer Medium Production web apps
DBSC (Token Binding) Yes Good Hardware-bound High (new spec) Future browsers
BFF Pattern Yes Good Tokens never reach client Medium-High SPAs with OAuth
mTLS Yes (cert revocation) Good Certificate-bound High Service-to-service

Which Model Should You Choose?

  • Building a traditional web app? Start with server-side sessions. Simple, secure, battle-tested.
  • Building an SPA that needs OAuth tokens? Use the BFF pattern. Don't store tokens in localStorage.
  • Need stateless verification at scale? Use JWT + server-side hybrid. Short-lived access tokens + refresh rotation.
  • Service-to-service communication? Use mTLS or signed JWTs with short expiry.
  • Want the strongest browser session protection? Watch DBSC development. Deploy when available.
  • Whatever you choose: Use httpOnly cookies, short token lifetimes, and a server-side revocation mechanism.

Real-World Impact

Session cookie theft isn't theoretical - it's the #1 technique behind major breaches today.

Lumma Stealer Campaign (2025)

Lumma infostealer distributed via fake CAPTCHA pages and cracked software. Steals browser cookies, passwords, and crypto wallets. Microsoft disrupted its infrastructure in May 2025 after it infected 394,000+ Windows machines.

Snowflake Customer Breaches (2024)

Attackers used stolen session tokens (harvested by infostealers) to access Snowflake accounts of 165+ organizations including Ticketmaster (560M records) and AT&T (110M records). No MFA bypass needed - the cookies were enough.

Dutch Police Token Theft (2024)

A state actor stole session cookies from a Dutch police officer's machine via infostealer malware, accessing contact details of 63,000 officers. The attacker bypassed all authentication including the police network's MFA.

Cookie-Bite Attack on Azure Entra (2025)

Researchers demonstrated a browser extension that exfiltrates Azure Entra ID session cookies in real-time, bypassing MFA and Conditional Access Policies to access Microsoft 365 services including email, Teams, and SharePoint.

This lab demonstrates the exact techniques used in these attacks.
