🕷 Session Replay Attack - Hardened

Paste a stolen session token below. The hardened app will show you exactly which security controls block the replay attempt and why.

Cross-app note (port 3001 vs 3002): The vulnerable and hardened apps use completely different JWT signing secrets. A token from port 3001 will fail verification on port 3002. If you appear logged in on both, those are independent sessions. Both apps use the same cookie name (session) on localhost, so logging into one may overwrite the other's cookie in your browser - use incognito windows to keep them separate.

To test cross-app replay: Copy a token from the vulnerable app's Cookie Vault, then paste it into the replay field below.

Browser Injection

Manual
Pro tip: Open this Attack Console in an incognito/private window to better simulate a real attacker. Incognito mode starts with no cookies - you'll only have the stolen token, just like an attacker on a separate machine.

Chrome: Ctrl+Shift+N (Mac: Cmd+Shift+N)  ·  Firefox: Ctrl+Shift+P (Mac: Cmd+Shift+P)  ·  Safari: Cmd+Shift+N
1
Open DevTools - press F12 or Cmd+Option+I (Mac) / Ctrl+Shift+I (Windows/Linux).
2
Navigate to cookies - go to the Application tab (Chrome/Edge) or Storage tab (Firefox). Expand Cookies and click on the current domain.
3
Add or edit the cookie - find (or create) a cookie named session. Paste the stolen JWT value into the Value field.
4
Refresh the page - press F5. The browser sends the injected cookie with the request - but the hardened server validates it against the session store, checks device binding, and verifies token expiry. The stolen cookie is rejected.
On the hardened app, manual injection still fails. Even if you successfully inject a stolen token into DevTools or curl, the server validates sessions against its server-side session store, checks IP and User-Agent binding, and enforces 15-minute token expiry. Defense-in-depth means no single bypass works.

Vulnerable vs Hardened - Side by Side

Vulnerable App (port 3001)

  • httpOnly: false - JS reads cookie
  • 7-day token expiry
  • No server-side session store
  • No device binding
  • sameSite: None
  • Logout only clears browser cookie
  • Stolen token works until expiry

Hardened App (port 3002)

  • httpOnly: true - JS gets nothing
  • 15-minute token expiry
  • Server-side session with revocation
  • IP + User-Agent binding
  • sameSite: Strict
  • Logout invalidates server session
  • Multiple layers block replay
