How Login Works

Every login flow - password, TOTP, OAuth - ends the same way: the server issues a session cookie. Click each step to learn more.

You - the user. You open a login page in your browser. You're about to prove your identity using one of many methods: a password, a TOTP code, a fingerprint, or Google OAuth. At this stage, you have no session - the server doesn't know who you are.
Credentials - what you provide. This could be a username + password, a 6-digit TOTP code from your authenticator app, or clicking "Sign in with Google." Each method proves your identity differently, but they all feed into the same next step: server-side verification.
Server Verifies - authentication happens here. The server checks your credentials against its database (password hash, TOTP secret, or OAuth token exchange with Google). If everything matches, you're authenticated. This is the only moment your password/2FA/OAuth matters.
Session Cookie Issued - the critical moment. After verification, the server creates a session token (usually a JWT) and sends it back as a cookie. From now on, the browser sends this cookie with every request. The server never asks for your password again - the cookie IS your identity.
Hotel key card analogy: Logging in is like showing your ID at the hotel front desk. The session cookie is the key card they hand you. Anyone who holds that key card can open your room - they don't need your ID.

{ "alg": "HS256", "typ": "JWT" }
alg - the signing algorithm. HS256 means HMAC-SHA256, a symmetric algorithm where the server uses a single secret key to both create and verify signatures.
typ - declares this is a JWT. Helps parsers identify the token format.

Decoded Payload: 
{
  "userId": "u_abc123",
  "username": "alice",
  "email": "alice@example.com",
  "authMethod": "password-totp",
  "iat": 1711500000,
  "exp": 1712104800
}
userId - unique identifier for the user account
username - the display name
email - account email address
authMethod - how the user logged in (password-totp or google-oauth)
iat - "issued at" - Unix timestamp of when this token was created
exp - "expires" - Unix timestamp of when this token becomes invalid
Signature
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV...<click to learn>
How the signature works:
The server takes the Base64URL-encoded header and payload, concatenates them with a dot, and runs HMAC-SHA256 using a secret key. The result is the signature.

HMAC-SHA256(base64url(header) + "." + base64url(payload), secret)

The signature verifies integrity, not confidentiality. Anyone can decode the header and payload - they're just Base64URL encoded, not encrypted. The signature only proves the token hasn't been tampered with.
Myth busted: JWTs are NOT encrypted - anyone can decode the header and payload. The signature only proves the token wasn't tampered with. Treat JWT contents as public information.
Explore it yourself: Paste this token into jwt.io to see the decoded parts interactively. You'll see "Invalid Signature" - that's expected because jwt.io doesn't have the server's secret key.

Cookie Flags Heatmap

Side-by-side comparison of how each app configures its session cookie. Red = insecure. Green = secure. Click any flag to learn more.

🍪 Vulnerable App (port 3001)

httpOnly false
httpOnly: false means JavaScript can read this cookie via document.cookie. Any script running on the page - including XSS payloads or browser extensions - can steal the session token. The vulnerable app intentionally leaves this off to demonstrate the risk.
secure false
secure: false means the cookie is sent over both HTTP and HTTPS. On an unencrypted HTTP connection, the cookie travels in plaintext and can be intercepted by anyone on the network (coffee shop WiFi, ISP, etc.).

Myth busted: The secure flag protects the cookie in transit (the wire), not at rest (the cookie store). An infostealer reads the cookie from disk - HTTPS doesn't help against that.
sameSite None
sameSite: None means the cookie is sent with all cross-site requests. A malicious site could trigger requests to this app and the browser would attach the session cookie - enabling CSRF attacks. This is the least restrictive setting.
maxAge 7 days
maxAge: 7 days means a stolen token remains valid for up to a week. That's a huge window for an attacker. Even if the user changes their password, the stolen token continues to work because sessions are stateless - the server only checks the JWT signature, not a session store.

🔒 Hardened App (port 3002)

httpOnly true
httpOnly: true prevents JavaScript from reading the cookie. document.cookie returns an empty string. This blocks XSS-based cookie theft and simple browser extension attacks. However, it does NOT stop infostealers that read the cookie database file directly from disk.
secure true
secure: true ensures the cookie is only sent over HTTPS connections. Combined with TLS, this protects the cookie from network-level eavesdropping. The cookie cannot be intercepted on the wire.

Remember: This protects transmission, not storage. A disk-based infostealer bypasses this entirely.
sameSite Strict
sameSite: Strict means the cookie is only sent with same-site requests. If you click a link to this app from another site, the cookie won't be included in the first request. This effectively eliminates CSRF attacks.
maxAge 15 minutes
maxAge: 15 minutes drastically limits the replay window. Even if a token is stolen, it's only valid for a short time. The hardened app uses refresh token rotation - a new access token is issued every ~13 minutes, and each refresh token can only be used once. If an attacker tries to use an old refresh token, all sessions are invalidated.

Stage 1: Victim logs in legitimately.
The victim enters their username, password, and TOTP code - or clicks "Sign in with Google." The server verifies their identity. At this stage, everything is secure. The authentication step worked exactly as designed.

Attacker's view: Nothing yet. The attacker doesn't need to be present during login.
Defenses here: MFA, OAuth, and strong passwords all protect this step - but they're about to become irrelevant.
Stage 2: Cookie stored in the browser.
The server sends a Set-Cookie header. The browser stores the session token in its cookie database - an SQLite file on disk. From now on, every request to this server includes the cookie automatically.

Attacker's view: Still nothing - but the attack surface just appeared. The cookie now exists on disk.
Defenses: httpOnly prevents JS access. But the on-disk file is still readable by any process with file system access.
Stage 3: Infostealer malware executes.
Delivered via phishing, a trojanized download, or a malicious browser extension. The malware reads the browser's cookie database file directly from disk. This typically takes less than a second.

Two attack vectors:
Vector A: Disk-based theft
Reads the SQLite cookie database file. Bypasses httpOnly entirely - httpOnly only restricts JavaScript, not file system access.
Vector B: JavaScript theft
Uses document.cookie via XSS or malicious extension. Blocked by httpOnly on the hardened app.
Stage 4: Cookie exfiltrated to attacker's C2 server.
The stolen cookies are sent to the attacker's command-and-control (C2) server. This happens in seconds, often before the user notices anything. The data package typically includes: session cookies, stored passwords, browser history, and autofill data.

Attacker's view: They now have the exact same session token that the victim's browser is using.
Defenses: Short token expiry limits the window. Device binding means the token may fail from a different IP/User-Agent. But on the vulnerable app - 7 days and no binding.
Stage 5: Attacker replays the stolen cookie.
The attacker injects the stolen session token into their own browser (or uses curl/Postman). The server receives the cookie, checks the JWT signature - it's valid - and grants full access. No password. No 2FA. No OAuth. Just the cookie.

Attacker's view: Full access to the victim's account. They ARE the user as far as the server is concerned.
Defenses (hardened app): Server-side session check fails (unknown session ID), device binding fails (wrong IP/UA), and the token likely expired within 15 minutes. Multiple layers block replay.

Why Strong Auth Doesn't Help

Authentication and sessions are two different phases. Click each zone to understand the handoff.

→

The Handoff

🍪 Session

Cookie-based access happens here

The Authentication zone is where identity is verified. This is where strong security lives: passwords are hashed, TOTP codes are time-limited, OAuth delegates to Google. Multi-factor authentication (MFA) makes this zone very strong - an attacker can't easily fake your identity here.

But here's the problem: authentication only happens once per session. After that, the server relies entirely on the session token.
The Handoff - the critical moment.
After successful authentication, the server creates a session token and sends it as a cookie. This is where all your authentication effort gets compressed into a single string. The token carries your identity forward - whoever holds it IS you.

The core insight: Authentication protects the login. It does NOT protect the session. These are different security boundaries, and infostealers attack the session, not the login.
The Session zone is where the user spends 99.9% of their time. Every page load, every API call, every action sends the session cookie. The server checks: "Is this a valid, non-expired token?" If yes, access is granted. It never re-checks the password or 2FA code.

This is the attacker's target. They skip the authentication zone entirely and operate in the session zone with a stolen cookie.

Defense in Depth

Five layers of defense, stacked. Each layer blocks attacks that bypass the one below it. Toggle between perspectives.

Layer 5 📋 Audit Logging
Audit Logging records every session event: logins, logouts, token refreshes, and - critically - replay attempts. It doesn't prevent attacks, but it enables detection and incident response.

What it blocks: Nothing directly - it's a detection control.
What it enables: SOC/SIEM alerts on anomalous session activity, forensic analysis after a breach.
Analogy: Security cameras don't stop a break-in, but they let you catch the intruder and understand what happened.
Layer 4 🔏 Device Binding
Device Binding ties the session to the specific device that logged in, using the client's IP address and User-Agent string. If a stolen token is replayed from a different machine, the server detects the mismatch and rejects the request.

What it blocks: Session replay from a different network/device.
What it doesn't block: Replay from the same network (e.g., malware on the victim's machine replaying locally). Also, VPNs can make IP binding unreliable.
Analogy: Like a parking permit that includes your license plate number - the permit only works on your car.
Layer 3 🗃 Server-Side Store
Server-Side Session Store keeps a record of active sessions on the server. Each session has a unique ID that must exist in the store for the JWT to be valid. This enables immediate revocation - logging out actually invalidates the token, unlike stateless JWTs.

What it blocks: Session use after logout, revoked sessions, tokens from deleted accounts.
What it doesn't block: Replay with a currently valid, non-revoked session from a matching device.
Analogy: Like a guest list at a building entrance - having a badge isn't enough; your name must be on the active list.
Layer 2 ⏰ Short Expiry
Short Token Expiry (15 minutes) limits the window of opportunity for an attacker. Even if they steal a token, it becomes useless quickly. Combined with refresh token rotation: a new access token is issued every ~13 minutes. Each refresh token is single-use - if an attacker tries to reuse an old one, all sessions for that user are invalidated.

What it blocks: Attacks that take longer than 15 minutes to execute (which is most real-world scenarios).
What it doesn't block: Automated, near-instant replay attacks within the 15-minute window.
Analogy: Like a visitor pass that expires after 15 minutes - the guard won't let you in with yesterday's pass.
Layer 1 🔒 httpOnly
httpOnly prevents JavaScript from accessing the cookie via document.cookie. This blocks XSS attacks and malicious browser extensions from reading the session token. It's the first line of defense and the easiest to implement.

What it blocks: JavaScript-based cookie theft (XSS, browser extensions, injected scripts).
What it doesn't block: Disk-based theft. Infostealers read the SQLite cookie database file - httpOnly doesn't protect file system access. This is why defense in depth matters.
Analogy: Like a "staff only" sign on a door - it stops casual visitors but not someone with a master key.

Beyond This Lab

Emerging and advanced defenses that go beyond traditional session management.

🔒 DBSC (Device Bound Session Credentials)

Chrome proposal to cryptographically bind sessions to hardware.

DBSC uses the device's TPM (Trusted Platform Module) to create a key pair. The session is bound to the private key, which never leaves the hardware. A stolen cookie is useless without the private key.

DBSC Chrome Proposal

🔗 Token Binding

Cryptographically binds tokens to TLS connections.

Token Binding (RFC 8471) ties authentication tokens to the TLS connection. Even if a token is stolen, it can't be replayed on a different TLS connection. Requires browser and server support.

RFC 8471 - Token Binding Protocol

🔑 FIDO2 / Passkeys

Phishing-resistant, device-bound authentication.

FIDO2 and Passkeys use public-key cryptography tied to a specific device. There's no password to steal and no cookie to replay during authentication. However, the application still needs to issue a session after FIDO2 auth - that session can still be stolen. FIDO2 solves the authentication problem, not the session problem.

FIDO Alliance - FIDO2

🎯 Zero Trust

Continuous verification, never trust the session alone.

Zero Trust architectures continuously evaluate risk signals: device health, network location, user behavior patterns, and more. Instead of trusting a session token for its entire lifetime, every request is evaluated against current risk. Anomalous behavior triggers step-up authentication or session termination.

NIST SP 800-207 - Zero Trust Architecture
