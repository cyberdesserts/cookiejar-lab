Browser Injection

Manual
Pro tip: Open this Attack Console in an incognito/private window to better simulate a real attacker. Incognito mode starts with no cookies - you'll only have the stolen token, just like an attacker on a separate machine.

Chrome: Ctrl+Shift+N (Mac: Cmd+Shift+N)  ·  Firefox: Ctrl+Shift+P (Mac: Cmd+Shift+P)  ·  Safari: Cmd+Shift+N
1
Open DevTools - press F12 or Cmd+Option+I (Mac) / Ctrl+Shift+I (Windows/Linux).
2
Navigate to cookies - go to the Application tab (Chrome/Edge) or Storage tab (Firefox). Expand Cookies and click on the current domain.
3
Add or edit the cookie - find (or create) a cookie named session. Paste the stolen JWT value into the Value field.
4
Refresh the page - press F5. The browser sends the injected cookie with every request. The server accepts it as a legitimate session - you have now hijacked the victim's account without ever knowing their password.

How Session Cookie Theft Works in the Real World

1
Victim logs in legitimately - enters password, completes 2FA, or uses Google OAuth. The server issues a session cookie (JWT).
2
Infostealer malware runs - delivered via phishing email, malicious download, or trojanized software. It reads the browser's cookie database file directly from disk.
3
Cookies exfiltrated - the malware sends stolen cookies to the attacker's C2 server. This happens in seconds, often before the victim notices anything.
4
Attacker replays the cookie - they inject the stolen session token into their own browser (or use tools like curl/Postman). The server sees a valid JWT and grants access.
5
Authentication completely bypassed - the attacker never knew the password, never had the TOTP secret, never went through Google OAuth. The session token IS the identity.
